## Context
Plumbline is release-please-managed (the `VERSION` file carries the `x-release-please`
markers). The user repeatedly asks whether the published version is current, whether the
README's numbers/claims are accurate, and whether the `plumbline update` terminal CLI works.
This skill answers all three with evidence, not assertion.

## Your Task
Run a fast, evidence-based currency + consistency check and report honestly.

### Steps
1. **Sync state.** `git status -sb` (is local `main` == `origin/main`?), then read `VERSION`.
2. **Latest published release.** `gh release list -L 3` and compare the top tag to `VERSION`.
3. **Dogfood the update CLI.** `./config/claude/bin/plumbline update --check` → expect
   `status: up-to-date`, else report the delta. Also `./config/claude/bin/plumbline doctor`
   (PATH discoverability + resolved update slug).
4. **Claim consistency.** Confirm the README agent count is **derived, not hand-asserted**:
   run `bash config/claude/tests/test_readme_honesty.sh`. Confirm no test/fixture pins a
   version literal (counts/versions must read from `VERSION` or be derived).
5. **Report.** A short table — VERSION · latest release · CLI verdict · claim-drift status —
   each row with `file:line` or command evidence; mark anything you could not verify `MISSING`.

### Guardrails
- **Verify the CI *conclusion*, never just `mergeable`/`CLEAN`,** before calling a release good
  (a release-please branch can get no CI run at all).
- **No hardcoded version numbers in tests/fixtures** — a pinned literal is a time-bomb that
  fails on the next release; read `VERSION` dynamically or assert consistency.
- Don't claim "up to date" without the `plumbline update --check` evidence; if the CLI/network
  is unavailable, say `MISSING`, don't guess.
